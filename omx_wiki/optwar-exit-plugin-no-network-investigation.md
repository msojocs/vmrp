## Update (2026-06-28T13:25:30+08:00)

Resolved. Root cause was stale GOT snapshot restoration into a recycled private-child RW base. Disassembly and diagnostics refined the failing branch: `verdload.ext` is the 20916-byte child loaded at runtime base `0x2EEAF0`; `0x2F085C` (file `0x1D6C`) calls `0x2F2930` (file `0x3E40`) and treats return `0` as immediate failure, while setup at `0x2F3370` initializes low downloader state. The observed bad state was not a low-table call: a `hook_low_zero` diagnostic produced no relevant hits. Instead, after `verdload` was synced, its own disassembly-proven bridge-copy layout was correctly repaired at `rw+0x1F4..0x238`, but later global GOT snapshot restore wrote older self-pointer bridges back into the same shared RW low mutable window (`rw+0x44=0x10068`, `rw+0x48=0x1000C`, `rw+0x4C=0x10010`). That made the `0x2F3370` path skip the runnable-state initialization; `rw+0x38` stayed `0xFFFFFFFF`, `0x1D6C` returned `-1`, and the child drew the red failure UI before any network bridge.

Fix in `src/arm_ext_executor.c`: add a generic `arm_ext_should_skip_got_snapshot_restore()` guard. It uses the current RW owner and the existing Thumb bridge-copy layout proof; if the child proves its bridge table lives in a safe high window, the low `R9+0x38..0x7f` range is treated as mutable state and stale snapshot self-pointers are not restored there. This is data-driven, not app-name-specific, and it does not add fallback behavior. Kept diagnostic run `/tmp/skyengine-e2e-MXr2P5` showed `DIAG got_restore_skip` for `rw+0x44..0x7C`, then post-confirm network: `my_initNetwork(NULL, 'cmnet')`, async connect, and `my_send ...
