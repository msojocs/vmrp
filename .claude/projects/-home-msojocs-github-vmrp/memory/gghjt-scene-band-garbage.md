## 第八轮（2026-06-21，重大修正）：真因=platEx(1014) 拓展内存未实现，**前七轮"4004/res.bin"方向是错的场景状态**

**仍未解(下一步)** —— band 视觉**仍花屏**，已精确定位到“瓦片读入但未解码合成”：

### 第八轮·续2（2026-06-21 晚）：**决定性根因——调色板解码把"源位图"取成了屏幕缓冲**

继续逐层追到底，band 花屏的**直接根因已确定**：
- band 背景**确实是合成出来的**：bg 缓冲 0x6F1EF0(=exram+0xBF04, 由 0x65cf38 经 0x646c88 分配, 大小=272×270×2=146880=0x23DC0, 字面量 0x23DC0 不是 handle 而是 size) 被 **0x65D272 `strh`** 逐像素写入 40000+ 次填满，band draw(0x6610D7→0x661098→_DrawBitmap, mw=240,sx 循环)再贴屏。**所以早前"band 源=cg 过场图"是 dump 时序误判**；bg 缓冲实际内容=合成结果=绿色横纹。
- **合成用的是调色板解码**(0x65D1FC / 循环 0x65D266)：`dest[i] = r5[ r7[i] ]`，r5=调色板, r7=r5+r6=索引。
- **致命点**：`r5 = 调用 0x666f8c() 的返回值 = 0x2001BC = 屏幕缓冲！** 0x666f8c 返回 `[[G+0x38]+0x16C][0]` = 游戏的"当前源位图"，本应是**解压后的瓦片数据(调色板+索引)**，却是**屏幕**。于是 `dest[i]=screen[ screen[off+i] ]` → 拿屏幕像素当调色板/索引 → 绿色花屏。
- 解码态 r4(栈) 即 image struct：抓到正在解码 "3000"(47×72,data@+0x10=**0x8 无效!**)、"3001"(105×66,data=0x6f1ef0=bg 缓冲)、"cg2"(240×94,data=0x6e5fb8)。[r4+0x10]=**目标**缓冲(由 0x65d368 按 [r4+0xc] 字节数分配；"3000" 返回 0x8=分配失败)。即除了源位图错，"3000" 瓦片的目标缓冲分配也失败。

**两条可下手的线（下一轮）**：
1. **源位图错(主因)**：为什么 0x666f8c(当前源位图)=屏幕而非解压瓦片。0x666f8c 读游戏全局 `O=[G+0x38]`(G≈0x6460E8 区, sb/R9=0x66D5AC primary 正确)的 +0x16C。需 watch 写 `O+0x16C`(当前源位图指针)的代码：游戏在解压瓦片后应把它设成瓦片临时位图，定位为何停留在屏幕(疑与"设绘制目标/源位图"的机制有关——另注意 slot91(table[91]=&mr_screenBuf) draw-target 始终=屏幕 0x2001BC，游戏重定向绘制目标我方未跟踪，可能同源)。
2. **目标分配失败**：0x65d368 给 "3000" 返回 0x8。查 0x65d368 分配器在该尺寸/时机为何失败。

### 第八轮·续3：再细一层——**调色板基址没跳过瓦片头**
- 追 setter/getter：**0x666fa0(源位图 setter) 从未被调用**，只有 0x666f8c(getter) 反复被调 → 当前源位图永远是默认的**屏幕位图**(.p=0x2001BC)。游戏把瓦片当 scratch 解压进屏幕缓冲，再从中读调色板。
- 在解码处 dump r5(=0x2001BC) 头 12 字节 = **`2f 00 48 00 10 09 00 e9 8b 1f f8 e7`**：开头 `2f 00 48 00` = **w=47,h=72**——正是 "3000" 瓦片的原生**头**！即 0x2001BC 缓冲里放的是**带头的原生瓦片**。
- 解码 `dest[i]=r5[ idx[i]*2 ]`，r5=缓冲首址=**头的位置**，idxoff=0x12(9 个调色板项)。**于是头字节(w,h…)被当成头几个调色板色** → 颜色全错 → 绿屏。正确应是 r5 跳过瓦片头(原生头 ≈4~8 字节)再做调色板基址，或解压后的 scratch 本不含头。
- 结论：**根因 = 源位图取成屏幕(含带头的原生瓦片) + 调色板基址未跳头**。需定位游戏本应"解压瓦片到无头 scratch 并设为源位图"的步骤为何缺失/退化成屏幕。下一轮：watch 写 0x2001BC(谁把原生 "3000" 写进屏幕缓冲)，以及 O+0x16C 源位图指针为何不被设置。

### 第八轮·续4：纠正——源位图取屏幕是正常的；真正差异在"瓦片缓冲分配失败"
- **重要纠正**：cg2(240×94 过场图)**也**走同一解码、palette 也=0x2001BC(屏幕 scratch，含自己的头 `f0 00 5e 00`)，而 cg2 **解码正确**(其 data 缓冲重建=干净的两人物)。所以"源位图=屏幕含头"并非根因——游戏就是拿屏幕当解压 scratch，头被当前几个调色板项但无害。
- **真正差异**：每个瓦片解码到**自己的目标缓冲**([r4+0x10]，由 bump 分配器 `0x65d368` 从 arena 分配)：3001→0x6f1ef0、cg2→0x6e5fb8(均 exRam 内、有效)；但 **"3000"→0x8(无效)**。`0x65d368` 是 arena bump 分配器(r0=arena{base@0,off@4,cap@8})，3000 那次 **arena base≈8**(garbage)→返回 8。即 3000(及同类楼房瓦片)**目标缓冲分配失败**→该瓦片没被解码→拼进 bg 时该处是垃圾→band 花屏。
- 试过 exRam fresh-alloc(两块独立缓冲) vs 幂等(一块)：band 像素**都不变**——2 次 platEx 1014 不是双缓冲冲突。
- **未解的下一层**：为什么 3000 那次的 arena(`[sp+0x50]`) base≈8。arena 由上层分配/初始化；base=8 像是某次子分配返回了无效小值(级联失败)。可能与"解压 scratch=屏幕、屏幕双缓冲未实现(MR_SCREEN_FIRST/SECOND_BUF)导致合成期间 scratch 被显示刷新破坏"有关，但未证实。

**本轮停止点（已极深，远超前七轮，但仍未收敛到单一可改点）**：花屏链= exRam(已修)→背景合成运行→部分楼房瓦片目标缓冲分配失败(0x65d368 arena base≈8)→bg 缺片/垃圾。需要真机/参考实现对照 arena 初始化与 scratch/双缓冲语义，纯黑盒到此。已留全部地址与脚手架说明。

### 第八轮·续5（最终，无参考下黑盒到顶）：arena=bg 缓冲本身 + 解码源=屏幕只在 intro 才对
- arena 分配器 0x65d368 实测：`r0(arena)=0x66E94C`(=**band bg 层结构**)、`base=0x6F1EF0`(=band draw 的源!)、`cap=0x23DC0`(272×270×2)。4 次分配序列：① 0x1A70(=3000,47×72) off0 → ② 0x3624(=3001,105×66) → ③ **arena 复位 off=0** → 0xB040(=cg,240×94) → ④ 0xB040(cg)。即**楼房瓦片(3000/3001)先解进 bg 缓冲，随后 arena 复位、cg 覆盖**。band 画的就是这块"解码 scratch/arena"，不是独立的二维合成图 → 画出来是各瓦片顺序排布的横纹。
- 前述"3000 dest=0x8"与此处"base=0x6F1EF0 有效"的矛盾说明存在**多条解码路径/多个 arena**(不同时机 base 不同)，未完全对齐。
- **关于"解码源=屏幕"**：解码 `dest[i]=screen[idx[i]]`，源=0x666f8c()=屏幕。实测解码瞬间屏幕 0x2001BC **确实放着该瓦片原生字节**(2f 00 48 00=3000 头)——即游戏把瓦片当 scratch 写进屏幕再回读。cg2 之所以早前看着是干净的，是它在 **intro 期**解码(屏幕正好=cg)；楼房在**游戏期**解码，屏幕=UI/或瓦片不完整 → 回读出错。源位图 setter 0x666fa0 **全程未被调用**，故源恒=默认屏幕位图——真机也不调，说明真机靠"屏幕此刻=源"，疑我方**单屏缓冲/无双缓冲(MR_SCREEN_FIRST/SECOND_BUF)** 使合成期 scratch 与显示互相破坏。

**最终结论（无参考，纯黑盒到顶）**：花屏是多层级联——exRam(已修) → 背景用"屏幕当解压 scratch + arena=bg 缓冲"的合成管线 → 在我方单屏缓冲模型下，合成期屏幕 scratch 内容/arena 复位时序与真机不一致 → bg 缓冲最终是错配/被覆盖的解码输出 → band 横纹花屏。**单一可改点未定**；最可能的真机差异是**屏幕双缓冲未实现**(mythroad MR_SCREEN_FIRST_BUF/SECOND_BUF，见 mythroad_mini.c:1441+)。下一轮若继续：实现双缓冲、或让 0x666f8c 的"源位图"在解码期指向真正的瓦片数据而非可见屏幕。

## 根因链（已逐层反汇编定位）
1. 该 band 全部由游戏自带的**带 transform 的精灵 blit** 函数 `0x6486B0`（类 mr_drawRegion，dispatch on transform 0xb0/0xf0 快路径）绘制；store 在 `0x648B8E`/`0x648B3A`（`strh r6,[r0]`）。dest stride 正确=480B(240px)。
